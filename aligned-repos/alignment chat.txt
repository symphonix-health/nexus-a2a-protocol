11. Governance workflow example
Delegating uk.health
Root Authority
   → verifies UK operator identity
   → issues signed delegation for uk.health
   → publishes trust anchor
   → UK GHARRA becomes authoritative for uk.health
Registering patient-registry.nhs.uk.health
NHS operator
   → publishes AgentCard
   → submits to NHS GHARRA
   → NHS GHARRA validates
   → record becomes authoritative
   → peer registries receive signed update
BulletTrain discovery
BulletTrain
   → resolve patient-registry.nhs.uk.health
   → local GHARRA validates uk.health delegation
   → validates nhs.uk.health trust chain
   → returns agent endpoint
   → Nexus connects
12. Why this model is strong

It avoids three common failures:

Failure A — centralised bottleneck

One registry for the whole world would be politically and operationally fragile.

Failure B — vendor capture

If BulletTrain or Nexus owns the root, external adoption becomes harder.

Failure C — weak trust inheritance

Without signed delegation and trust anchors, name resolution is not enough for healthcare.

13. Strategic conclusion

The right answer is:

GHARRA owns the .health namespace and trust governance

Nexus-A2A consumes GHARRA outputs for secure routing

BulletTrain consumes GHARRA outputs for policy-aware orchestration

So your three repositories already map naturally to the three critical infrastructure layers.
A fourth repository is not required for function, though you may later want one for ecosystem-wide deployment artefacts and governance documents.
