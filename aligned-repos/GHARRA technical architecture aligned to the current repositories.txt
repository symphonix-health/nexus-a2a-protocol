# GHARRA technical architecture aligned to the current repositories

## 1. Alignment principle

The three repositories should map cleanly like this:

| Repository             | Existing/confirmed role                                                               | What GHARRA must provide                                       |
| ---------------------- | ------------------------------------------------------------------------------------- | -------------------------------------------------------------- |
| **BulletTrain**        | SignalBox orchestration already expects an `AgentRegistry` abstraction. ([GitHub][1]) | A real GHARRA client implementation behind that abstraction    |
| **Nexus-A2A-protocol** | Already enforces cert-bound tokens and has routing/stream scale-profile concerns.     | Trust metadata and routable agent endpoint records from GHARRA |
| **GHARRA**             | Global registry/trust repo                                                            | Namespace, discovery, trust anchors, federation, policy        |

So the architecture should not invent a new centre. It should **replace BulletTrain’s abstract AgentRegistry with GHARRA**, and make Nexus consume GHARRA’s trust and address metadata.

## 2. GHARRA logical services

Inside **`global-agent-registry`**, GHARRA should be split into six services or modules.

### A. Namespace Service

Owns:

* `.health` root namespace
* delegated zones such as `uk.health`, `ke.health`, `gh.health`
* name validation
* canonicalisation

### B. Registry Service

Owns:

* agent records
* AgentCard ingestion
* exact-name resolution
* capability search

### C. Trust Service

Owns:

* JWKS metadata
* mTLS CA bundles
* trust anchors by zone
* revocation

This is necessary because Nexus already expects cert-bound identity semantics.

### D. Federation Service

Owns:

* inter-registry synchronisation
* signed updates
* zone replication
* conflict resolution

### E. Policy Service

Owns:

* jurisdiction rules
* PHI export constraints
* allowed regions
* protocol admission constraints

This aligns with BulletTrain’s documented gateway security expectations. ([GitHub][1])

### F. Audit and Governance Service

Owns:

* delegation events
* trust changes
* record activation/deactivation
* compliance logging

## 4. Core GHARRA data models

These are the minimum data models GHARRA needs so BulletTrain and Nexus can consume them without ambiguity.

### AgentRecord

```json
{
  "name": "patient-registry.nhs.uk.health",
  "agent_id": "agnt_01...",
  "zone": "nhs.uk.health",
  "status": "active",
  "transport": {
    "protocol": "nexus-a2a",
    "endpoint": "https://router.nhs.uk/a2a/patient-registry",
    "protocol_versions": ["1.0", "1.1"],
    "feature_flags": ["routing.v1", "stream.resume.v1"]
  },
  "capabilities": [
    "FHIR.Patient.read",
    "identity.match"
  ],
  "authentication": {
    "mtls_required": true,
    "jwks_uri": "https://auth.nhs.uk/.well-known/jwks.json",
    "cert_bound_tokens_required": true
  },
  "policy_tags": ["phi", "uk-only"],
  "last_verified_at": "2026-03-14T12:00:00Z"
}
```

That `transport.feature_flags` field is deliberately aligned to the Nexus scale profile concepts already present. ([OpenAI Platform][2])

### ZoneRecord

```json
{
  "zone": "uk.health",
  "authoritative_registry": "https://gharra.uk.health",
  "delegated_by": ".health",
  "jwks_uri": "https://gharra.uk.health/.well-known/jwks.json",
  "status": "active"
}
```

### TrustRecord

```json
{
  "zone": "nhs.uk.health",
  "jwks_uri": "https://trust.nhs.uk/.well-known/jwks.json",
  "mtls_ca_bundle_uri": "https://trust.nhs.uk/ca.pem",
  "thumbprint_policy": "cnf.x5t#S256",
  "status": "active"
}
```

This directly aligns to Nexus’ existing thumbprint enforcement.

### PolicyRecord

```json
{
  "zone": "uk.health",
  "rules": {
    "phi_export_default": "deny",
    "allowed_regions": ["UK", "EU"],
    "mtls_required": true,
    "signed_agentcard_required": true
  }
}
```

## 5. GHARRA API surface

These endpoints should be implemented in GHARRA now, because they map directly to BulletTrain and Nexus needs.

### Resolution

```text
GET /v1/agents/{name}
```

Purpose:

* BulletTrain SignalBox resolves an agent by canonical name.

### Discovery

```text
GET /v1/discovery?capability=...
```

Purpose:

* BulletTrain builds dynamic workflows by capability rather than by hardcoded endpoint.

### Registration

```text
POST /v1/agents
POST /v1/agents/register
```

Purpose:

* Nexus/A2A agents publish AgentCards and become discoverable.

### Zones

```text
GET /v1/zones
GET /v1/zones/{zone}
POST /v1/zones/{zone}/delegate
POST /v1/zones/{zone}/revoke
```

Purpose:

* GHARRA root and delegated authorities manage namespace structure.

### Trust

```text
GET /v1/trust/{zone}
POST /v1/trust/{zone}/anchors
POST /v1/trust/{zone}/revoke
```

Purpose:

* Nexus transport can verify target trust material.

### Federation

```text
GET /v1/federation/peers
POST /v1/federation/sync
POST /v1/federation/update
```

Purpose:

* multi-registry GHARRA operation.

## 6. BulletTrain integration, aligned to what already exists

BulletTrain already documents an `AgentRegistry` role in SignalBox. ([GitHub][1])
So the correct refactor is not to invent a new orchestration concept. It is to **bind that existing abstraction to GHARRA**.

### Replace:

```python
AgentRegistry.validate(agent_id)
```

### With:

```python
gharra.resolve(name)
gharra.lookup(capability)
gharra.get_policy(zone)
```

### BulletTrain GHARRA client interface

```python
class GharraClient:
    def resolve_agent(self, name: str) -> dict: ...
    def lookup_by_capability(self, capability: str, region: str | None = None) -> list[dict]: ...
    def get_zone_policy(self, zone: str) -> dict: ...
    def get_trust_record(self, zone: str) -> dict: ...
```

### SignalBox flow

```text
SignalBox
  → GHARRA resolve/discovery
  → policy/trust check
  → Nexus transport call
  → remote agent
```

That also fits BulletTrain’s existing external gateway security expectations around mTLS, JWT audience, replay protection and idempotency. ([GitHub][1])

## 7. Nexus-A2A integration, aligned to what already exists

Nexus already has transport/auth machinery that checks certificate-bound tokens via `cnf.x5t#S256`.
So GHARRA should not duplicate transport auth logic. It should provide the **metadata Nexus needs to decide whether a route is valid**.

### Nexus should consume from GHARRA:

* `endpoint`
* `protocol`
* `protocol_versions`
* `feature_flags`
* `jwks_uri`
* `mtls_required`
* `policy_tags`

### Nexus admission rule

Before opening a route, Nexus should check:

1. Is the target name resolvable in GHARRA?
2. Is the zone delegated and active?
3. Is the trust record valid?
4. Does the target require cert-bound tokens?
5. Do policy tags allow the requested operation?

This is consistent with the existing Nexus security posture rather than a new security model.

## 8. AgentCard alignment

The public A2A direction is that agents publish **Agent Cards** for discovery. ([GitHub][3])
So GHARRA should use AgentCard ingestion as its registration input.

### AgentCard to GHARRA mapping

| AgentCard field            | GHARRA field                     |
| -------------------------- | -------------------------------- |
| agent name/card identifier | `name`                           |
| connection info            | `transport.endpoint`             |
| auth schemes               | `authentication`                 |
| skills/capabilities        | `capabilities`                   |
| protocol support           | `transport.protocol_versions`    |
| vendor-specific metadata   | `policy_tags` / extension fields |

### Required endpoint

```text
https://<agent-host>/.well-known/agent.json
```

### GHARRA registration flow

```text
Agent publishes AgentCard
   → GHARRA fetches/validates
   → creates AgentRecord
   → indexes by name and capability
   → federates update to peers
```

## 10. Eventing and federation

GHARRA should be event-driven internally.

### Internal event types

* `zone.delegated`
* `zone.revoked`
* `agent.registered`
* `agent.updated`
* `agent.revoked`
* `trust.anchor_added`
* `trust.anchor_revoked`
* `policy.updated`
* `federation.sync_completed`

### Why this aligns

It lets BulletTrain and Nexus react to changes without polling, and it preserves a full audit trail expected in healthcare systems.
